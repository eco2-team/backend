# 네이버페이 신입 개발자 지원 자기소개서

## [필수] '이 정도면 충분하다'는 생각 대신, 끝까지 몰입했던 가장 의미 있는 개발 경험

전직장에서는 Kubernetes 기반의 프로덕션 환경을 경험했지만, 단순히 기존 인프라를 사용하는 입장에서 내부 구조를 깊이 이해하지 못했습니다. "왜 이렇게 설계되어 있을까?"라는 호기심이 이 프로젝트의 시작이었고, 간단한 구성에서 시작해 프로덕션 수준의 Self-Managed Kubernetes 클러스터를 직접 구축하기로 결정했습니다. 초기 목표는 docker-compose로 빠르게 프로토타이핑하는 것이었지만, 멀티 노드와 고가용성 등 프로덕션의 핵심 요소를 재현하기에는 한계가 명확했습니다. 그래서 "이 정도면 충분하다"는 타협 없이 처음부터 Self-Managed Kubernetes 클러스터를 구축하고, 그 위에 API 서버 로직을 설계 및 배포하기로 결심했습니다.

가장 큰 난관은 의존성 문제였습니다. RabbitMQ에 메트릭 수집을 활성화했지만 ServiceMonitor CRD not found 오류가 발생했고, Prometheus Operator가 RabbitMQ보다 나중에 설치되어 CRD가 아직 생성되지 않았기 때문이었습니다. Ansible playbook의 실행 순서를 재조정하고, CRD 생성 대기 로직을 추가하면서 Kubernetes 리소스의 생성 순서와 타이밍의 중요성을 체득했습니다. 또 다른 문제는 PVC가 계속 Pending 상태였는데, 원인은 AWS EBS CSI Driver 미설치와 IAM 권한 부족이었습니다. kubectl describe pvc로 이벤트 로그를 확인하고, Terraform으로 EBS CSI Driver 설치 playbook을 추가하고, IAM 권한을 수정하는 과정에서 인프라 계층 간의 의존성을 이해하게 되었습니다.

완성도를 높이기 위해 auto-rebuild.sh 스크립트를 작성해 40-50분 걸리던 배포를 한 번의 명령으로 자동화했고, TROUBLESHOOTING.md에 모든 오류와 해결 방법을 상세히 기록해 문서화 습관을 기를 수 있었습니다. 그리고 클러스터 구축을 넘어 4-Tier Layered Architecture로 API 서버 로직을 설계했습니다. FastAPI 기반의 동기 API 서버와 Celery 기반의 비동기 워커를 분리하고, RabbitMQ를 통한 메시지 큐 패턴으로 확장 가능한 구조를 만들었습니다. Claude 4.5와 Cursor를 적극 활용해 복잡한 Terraform/Ansible 구문 오류를 빠르게 진단하고, 반복적인 YAML 작성 시간을 70% 감소시켰습니다.

이 프로젝트는 현재도 진행 중이며, 인프라 자동화를 완성하고 API 서버의 핵심 기능 구현에 집중하고 있습니다. 이 과정에서 배운 가장 큰 교훈은 "왜?"를 끝까지 파고드는 습관이었습니다. 단순히 에러 메시지를 검색해 해결하는 것이 아니라, 왜 그 에러가 발생했는지 근본 원인을 파악하는 과정에서 시스템 전반을 이해하게 되었습니다. 또한 Terraform과 Ansible로 모든 인프라를 코드화하니 클러스터를 여러 번 파괴하고 재생성해도 동일한 결과를 얻을 수 있어, "실패해도 되니까 시도해보자"는 마음가짐을 갖게 되었습니다. Claude 4.5와 Cursor는 생각의 속도를 높여주는 도구였지만, AI가 생성한 코드를 그대로 사용하지 않고 "이 코드가 왜 동작하는가?"를 항상 검증하는 과정에서 진정한 학습이 이루어졌습니다. 간단한 구성에서 시작해 Self-Managed Kubernetes로 발전시키고, Terraform과 Ansible로 완전히 자동화하고, API 서버 로직을 설계 및 배포하는 경험은 "이 정도면 충분하다"는 타협 없이 완성도를 높이는 것이 얼마나 가치 있는 일인지를 보여주었습니다. 앞으로 네이버페이에서 만나게 될 복잡한 금융 시스템의 인프라와 아키텍처를 이해하고 개선하는 데, 이 경험에서 얻은 근본 원인을 파악하는 사고력과 끝까지 몰입하는 집중력을 발휘하고 싶습니다.
