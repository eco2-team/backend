## 개요

NUMA (Non-Uniform Memory Access)는 **멀티프로세서 시스템에서 메모리 접근 지연 시간이 균일하지 않은 아키텍처**이다. 각 프로세서는 "가까운" 로컬 메모리에는 빠르게, "먼" 원격 메모리에는 느리게 접근한다.

### UMA vs NUMA

```
┌─────────────────────────────────────────────────────────────────┐
│                    UMA (Uniform Memory Access)                    │
├─────────────────────────────────────────────────────────────────┤
│                                                                  │
│         ┌─────┐  ┌─────┐  ┌─────┐  ┌─────┐                      │
│         │CPU 0│  │CPU 1│  │CPU 2│  │CPU 3│                      │
│         └──┬──┘  └──┬──┘  └──┬──┘  └──┬──┘                      │
│            │        │        │        │                          │
│         ═══╪════════╪════════╪════════╪════  System Bus         │
│                          │                                       │
│                     ┌────┴────┐                                  │
│                     │ Memory  │  모든 CPU가 동일한              │
│                     │ (공유)   │  지연시간으로 접근              │
│                     └─────────┘                                  │
│                                                                  │
│  지연 시간: 모든 CPU → Memory: 동일                              │
│                                                                  │
└─────────────────────────────────────────────────────────────────┘

┌─────────────────────────────────────────────────────────────────┐
│                    NUMA (Non-Uniform Memory Access)              │
├─────────────────────────────────────────────────────────────────┤
│                                                                  │
│     ┌───────────────────┐         ┌───────────────────┐         │
│     │     Node 0        │         │     Node 1        │         │
│     │  ┌─────┐ ┌─────┐  │   QPI   │  ┌─────┐ ┌─────┐  │         │
│     │  │CPU 0│ │CPU 1│  │◄───────►│  │CPU 2│ │CPU 3│  │         │
│     │  └──┬──┘ └──┬──┘  │   /     │  └──┬──┘ └──┬──┘  │         │
│     │     │       │     │  HT     │     │       │     │         │
│     │     └───┬───┘     │         │     └───┬───┘     │         │
│     │         │         │         │         │         │         │
│     │    ┌────┴────┐    │         │    ┌────┴────┐    │         │
│     │    │ Memory  │    │         │    │ Memory  │    │         │
│     │    │ (Local) │    │         │    │ (Local) │    │         │
│     │    └─────────┘    │         │    └─────────┘    │         │
│     └───────────────────┘         └───────────────────┘         │
│                                                                  │
│  지연 시간:                                                       │
│    CPU 0 → Node 0 Memory (Local):  ~50ns                        │
│    CPU 0 → Node 1 Memory (Remote): ~100ns (2배)                  │
│                                                                  │
└─────────────────────────────────────────────────────────────────┘
```

### NUMA 등장 배경

```
┌─────────────────────────────────────────────────────────────────┐
│                  메모리 대역폭 한계의 벽                          │
├─────────────────────────────────────────────────────────────────┤
│                                                                  │
│  문제:                                                           │
│  ─────                                                           │
│  • CPU 코어 수 증가 속도 > 메모리 대역폭 증가 속도               │
│  • 단일 메모리 컨트롤러가 병목                                    │
│  • 모든 CPU가 하나의 버스로 메모리 접근 → 경쟁                   │
│                                                                  │
│  해결: NUMA                                                      │
│  ────────────                                                    │
│  • 각 CPU 소켓에 전용 메모리 컨트롤러 배치                       │
│  • 로컬 메모리 접근은 독립적 → 대역폭 병목 해소                  │
│  • 원격 접근은 인터커넥트 통과 → 지연 증가                       │
│                                                                  │
│  ┌─────────────────────────────────────────────────────────┐    │
│  │                                                         │    │
│  │   "NUMA는 성능을 위해 균일성을 포기한 트레이드오프"     │    │
│  │                                                         │    │
│  └─────────────────────────────────────────────────────────┘    │
│                                                                  │
│  확장성 비교:                                                     │
│  ─────────────                                                   │
│  UMA (SMP): ~4-8 CPU까지 확장 가능                               │
│  NUMA: ~수천 CPU까지 확장 가능 (SGI UV 시리즈)                   │
│                                                                  │
└─────────────────────────────────────────────────────────────────┘
```

## 메모리 일관성 (Cache Coherence)

### 1. 캐시 일관성 문제

```
┌─────────────────────────────────────────────────────────────────┐
│                  캐시 일관성 문제 시나리오                         │
├─────────────────────────────────────────────────────────────────┤
│                                                                  │
│  초기 상태: Memory[X] = 0                                        │
│                                                                  │
│   CPU 0 (Node 0)              CPU 1 (Node 1)                    │
│   ───────────────             ───────────────                    │
│   Cache[X] = 0                Cache[X] = 0                       │
│        │                           │                             │
│        ▼                           │                             │
│   Write X = 100                    │                             │
│   Cache[X] = 100                   │                             │
│        │                           ▼                             │
│        │                      Read X                             │
│        │                      Cache[X] = 0  ← 문제!              │
│        │                           │                             │
│        └───────────────────────────┘                             │
│                                                                  │
│  문제: CPU 1이 CPU 0의 쓰기를 보지 못함                          │
│  해결: 캐시 일관성 프로토콜 (MESI, MOESI)                        │
│                                                                  │
└─────────────────────────────────────────────────────────────────┘
```

### 2. MESI 프로토콜

**Modified, Exclusive, Shared, Invalid**

```
┌─────────────────────────────────────────────────────────────────┐
│                    MESI 상태 전이도                               │
├─────────────────────────────────────────────────────────────────┤
│                                                                  │
│  ┌─────────────────────────────────────────────────────────┐    │
│  │                                                         │    │
│  │   ┌───────────┐                      ┌───────────┐     │    │
│  │   │  Invalid  │◄─────────────────────│ Modified  │     │    │
│  │   │    (I)    │    (다른 CPU 읽기)    │    (M)    │     │    │
│  │   └─────┬─────┘                      └─────┬─────┘     │    │
│  │         │                                  ▲           │    │
│  │         │ (로컬 읽기,                      │           │    │
│  │         │  메모리에서)                     │(로컬 쓰기) │    │
│  │         ▼                                  │           │    │
│  │   ┌───────────┐    (다른 CPU 읽기)   ┌─────┴─────┐     │    │
│  │   │ Exclusive │──────────────────────► │  Shared   │     │    │
│  │   │    (E)    │                      │    (S)    │     │    │
│  │   └───────────┘                      └───────────┘     │    │
│  │                                                         │    │
│  └─────────────────────────────────────────────────────────┘    │
│                                                                  │
│  상태 설명:                                                       │
│  ───────────                                                     │
│  M (Modified):  수정됨, 이 캐시만 유효, 메모리와 다름            │
│  E (Exclusive): 배타적, 이 캐시만 가짐, 메모리와 같음            │
│  S (Shared):    공유됨, 여러 캐시가 가짐, 읽기만 가능            │
│  I (Invalid):   무효, 캐시 라인 사용 불가                        │
│                                                                  │
└─────────────────────────────────────────────────────────────────┘
```

### 3. MOESI 프로토콜 (AMD)

**Modified, Owned, Exclusive, Shared, Invalid**

```
MOESI는 MESI에 Owned 상태를 추가:

O (Owned): 
  • 수정된 데이터의 소유자
  • 다른 캐시도 Shared로 가질 수 있음
  • 메모리에 Write-back 책임

장점:
  • Shared 상태에서 메모리 접근 감소
  • Cache-to-Cache 전송 가능
```

### 4. Directory-Based vs Snooping

```
┌─────────────────────────────────────────────────────────────────┐
│            캐시 일관성 프로토콜 구현 방식                          │
├─────────────────────────────────────────────────────────────────┤
│                                                                  │
│  Snooping (버스 기반):                                           │
│  ─────────────────────                                           │
│  • 모든 캐시가 버스의 모든 트랜잭션을 "감청" (snoop)             │
│  • 관련 캐시 라인이면 상태 업데이트                              │
│  • 장점: 구현 단순                                               │
│  • 단점: 버스 대역폭 병목, 확장성 제한 (~4-8 프로세서)           │
│                                                                  │
│  ┌─────────────────────────────────────────────────────────┐    │
│  │  CPU 0   CPU 1   CPU 2   CPU 3                          │    │
│  │    │       │       │       │                            │    │
│  │    └───────┴───────┴───────┘                            │    │
│  │              │                                          │    │
│  │    ══════════╪══════════════  (모두 감청)               │    │
│  │              │                                          │    │
│  │          Memory                                         │    │
│  └─────────────────────────────────────────────────────────┘    │
│                                                                  │
│  Directory-Based (디렉토리 기반):                                │
│  ────────────────────────────────                                │
│  • 각 메모리 블록에 대해 어느 캐시가 가지고 있는지 기록          │
│  • 무효화 시 해당 캐시에만 메시지 전송                           │
│  • 장점: 확장성 우수 (수백-수천 프로세서)                        │
│  • 단점: 디렉토리 저장 오버헤드, 간접 접근 지연                  │
│                                                                  │
│  ┌─────────────────────────────────────────────────────────┐    │
│  │  Memory Block      Directory Entry                      │    │
│  │  ─────────────     ────────────────────────────         │    │
│  │  Block 0     →     {Node 0, Node 2} (Shared)            │    │
│  │  Block 1     →     {Node 1} (Modified)                  │    │
│  │  Block 2     →     {} (Uncached)                        │    │
│  │  ...                                                     │    │
│  └─────────────────────────────────────────────────────────┘    │
│                                                                  │
│  현대 NUMA 시스템: Directory-Based 사용                          │
│                                                                  │
└─────────────────────────────────────────────────────────────────┘
```

### 5. False Sharing 문제

```
┌─────────────────────────────────────────────────────────────────┐
│                    False Sharing 문제                             │
├─────────────────────────────────────────────────────────────────┤
│                                                                  │
│  캐시 라인 크기: 64 bytes (일반적)                               │
│                                                                  │
│  ┌─────────────────────────────────────────────────────────┐    │
│  │              Cache Line (64 bytes)                      │    │
│  │  ┌───────────────────┬───────────────────┐              │    │
│  │  │   Variable A      │   Variable B      │              │    │
│  │  │   (CPU 0 사용)    │   (CPU 1 사용)    │              │    │
│  │  └───────────────────┴───────────────────┘              │    │
│  │                                                          │    │
│  │  CPU 0: A 수정 → 전체 캐시 라인 Modified                 │    │
│  │  CPU 1: B 읽기 → CPU 0에서 캐시 라인 가져와야 함         │    │
│  │  CPU 1: B 수정 → CPU 0의 캐시 라인 Invalidate            │    │
│  │  CPU 0: A 읽기 → CPU 1에서 캐시 라인 가져와야 함         │    │
│  │                                                          │    │
│  │  결과: 서로 다른 변수를 접근하지만 캐시 라인 핑퐁        │    │
│  │                                                          │    │
│  └─────────────────────────────────────────────────────────┘    │
│                                                                  │
│  해결 방법:                                                       │
│  ───────────                                                     │
│  1. 패딩으로 변수를 다른 캐시 라인에 배치                        │
│  2. `__cacheline_aligned` 속성 사용                              │
│  3. 데이터 구조 설계 시 접근 패턴 고려                           │
│                                                                  │
│  C 코드 예시:                                                     │
│  ────────────                                                    │
│  // 잘못된 예                                                     │
│  struct counter {                                                │
│      int count_a;  // CPU 0 사용                                 │
│      int count_b;  // CPU 1 사용                                 │
│  };                                                               │
│                                                                  │
│  // 올바른 예 (패딩 추가)                                         │
│  struct counter {                                                │
│      int count_a;                                                │
│      char padding[60];  // 64 - sizeof(int)                      │
│      int count_b;                                                │
│  };                                                               │
│                                                                  │
└─────────────────────────────────────────────────────────────────┘
```

## NUMA-aware 프로그래밍

### 1. 메모리 할당 정책

```
┌─────────────────────────────────────────────────────────────────┐
```

### 4. CPU Affinity 설정

```c
#define _GNU_SOURCE
#include <sched.h>
#include <pthread.h>

// CPU 집합 생성
cpu_set_t cpuset;
CPU_ZERO(&cpuset);
CPU_SET(0, &cpuset);  // CPU 0
CPU_SET(1, &cpuset);  // CPU 1

// 현재 스레드를 특정 CPU에 바인딩
sched_setaffinity(0, sizeof(cpuset), &cpuset);

// pthread 생성 시 affinity 설정
pthread_attr_t attr;
pthread_attr_init(&attr);
pthread_attr_setaffinity_np(&attr, sizeof(cpuset), &cpuset);
pthread_create(&thread, &attr, thread_func, NULL);
```

```
     ▲
  이상적│                          ╭─────────────  (이상적: 선형)
```

### 4. 해결 전략

```
┌─────────────────────────────────────────────────────────────────┐
│             확장성 향상을 위한 전략                               │
├─────────────────────────────────────────────────────────────────┤
│                                                                  │
│  1. 락 세분화 (Fine-grained Locking)                            │
│     ─────────────────────────────────                            │
│     • 하나의 큰 락 → 여러 개의 작은 락                           │
│     • 예: 해시 테이블 버킷별 락                                  │
│                                                                  │

│     ────────────────────────                                     │
│     • 읽기에 락 불필요                                           │
│     • 쓰기 시 복사본 생성 후 atomic swap                         │
│     • 읽기 빈도 >> 쓰기 빈도 시 효과적                           │
│                                                                  │
│  3. Per-CPU 자료구조                                             │
│     ─────────────────────                                        │
│     • 각 CPU가 독립적인 복사본 유지                              │
│     • 읽기/쓰기 시 락 불필요                                     │
│     • 주기적 통합 필요 (예: 카운터)                              │
│                                                                  │
│  4. 락 없는 알고리즘 (Lock-free)                                 │
│     ───────────────────────────                                  │
│     • CAS (Compare-and-Swap) 등 원자적 연산 사용                │
│     • 구현 복잡하지만 최고 성능                                  │
│                                                                  │
│  5. NUMA-aware 설계                                              │
│     ─────────────────                                            │
│     • 데이터를 접근하는 CPU 가까이 배치                          │
│     • 노드 간 통신 최소화                                        │
│                                                                  │
└─────────────────────────────────────────────────────────────────┘
```

## 논문 분석: "The Scalable Commutativity Rule"

### 1. 논문 개요

| 항목 | 내용 |
|------|------|
| **학회** | SOSP 2013 |
| **저자** | Austin T. Clements et al. (MIT) |
| **핵심 기여** | 확장성과 인터페이스 설계의 관계 규명 |

### 2. Scalable Commutativity Rule

```
┌─────────────────────────────────────────────────────────────────┐
│          Scalable Commutativity Rule (SCR)                       │
├─────────────────────────────────────────────────────────────────┤
│                                                                  │
│  핵심 정리:                                                       │
│  ───────────                                                     │
│  "연산들이 교환 가능하면 (commutative),                          │
│   충돌 없는 (conflict-free) 구현이 존재한다."                    │
│                                                                  │
│  수식:                                                           │
│                                                                  │
│     ops ∈ Commutative ⟹ ∃ Conflict-Free Implementation          │
│                                                                  │
│  예시:                                                           │
│  ───────                                                         │
│  • open("file", O_RDONLY) || open("file", O_RDONLY)             │
│    → 두 open이 동시에 호출되어도 결과 순서와 무관                │
│    → conflict-free 구현 가능                                    │
│                                                                  │
│  • creat("file") || creat("file")                               │
│    → 하나만 성공해야 하므로 순서가 중요                          │
│    → conflict 필연적                                            │
│                                                                  │
└─────────────────────────────────────────────────────────────────┘
```

### 3. 인터페이스 설계의 영향

```
┌─────────────────────────────────────────────────────────────────┐
│            인터페이스 설계가 확장성에 미치는 영향                  │
├─────────────────────────────────────────────────────────────────┤
│                                                                  │
│  잘못된 인터페이스 (확장성 저하):                                │
│  ──────────────────────────────                                  │
│                                                                  │
│  fd = open(path, flags);                                        │
│  • 파일 디스크립터 번호가 순차적 할당                            │
│  • "가장 작은 사용 가능한 fd" 반환                               │
│  • → 글로벌 fd 테이블 락 필요                                    │
│                                                                  │
│  개선된 인터페이스 (확장성 향상):                                │
│  ─────────────────────────────                                   │
│                                                                  │
│  fd = open(path, flags | O_CLOEXEC);                            │
│  • 임의의 fd 번호 허용 (명시적 요청이 아니면)                    │
│  • → per-CPU fd 할당 가능                                       │
│                                                                  │
│  핵심 통찰:                                                       │
│  ───────────                                                     │
│  "POSIX API의 일부 의미론이 근본적으로 확장성 제한"              │
│                                                                  │
└─────────────────────────────────────────────────────────────────┘
```

### 4. COMMUTER 도구

논문에서 개발한 도구로, 인터페이스의 commutativity를 자동 분석:

```
입력:  인터페이스 명세 (Python symbolic model)
출력:  어떤 연산 조합이 commutative한지 테스트 케이스

예시 결과:
• 13,664개의 syscall 조합 테스트
• 대부분 commutative → 확장 가능
• 일부 non-commutative → 확장성 한계 불가피
```
