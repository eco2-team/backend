## 개요

MapReduce는 **대규모 데이터 처리를 위한 프로그래밍 모델이자 런타임 시스템**이다.
개발자는 `map`과 `reduce` 두 함수만 정의하면, 병렬화, 분산, 장애 복구는 시스템이 자동으로 처리한다.

### 핵심 기여

```
┌─────────────────────────────────────────────────────────────────┐
│                  MapReduce의 핵심 기여                            │
├─────────────────────────────────────────────────────────────────┤
│                                                                  │
│  1. 단순한 프로그래밍 모델                                        │
│     → 함수형 프로그래밍의 map/reduce를 분산 시스템에 적용          │
│     → 개발자는 비즈니스 로직만 작성                                │
│                                                                  │
│  2. 자동 병렬화                                                  │
│     → 데이터 파티셔닝 및 태스크 스케줄링 자동화                    │
│     → 수천 대 머신에서 투명하게 실행                              │
│                                                                  │
│  3. 장애 복구                                                    │
│     → Worker 장애 시 자동 재실행                                  │
│     → Stragglers(느린 태스크) 중복 실행                           │
│                                                                  │
│  4. 로컬리티 최적화                                               │
│     → GFS와 통합, 데이터 위치에서 연산 실행                       │
│     → 네트워크 대역폭 절약                                        │
│                                                                  │
└─────────────────────────────────────────────────────────────────┘
```

## 프로그래밍 모델

### 1. Map 함수

**정의**: 입력 key-value 쌍을 중간 key-value 쌍 리스트로 변환

```
map(k1, v1) → list(k2, v2)
```

**수식 표현**:

\[
\text{Map}: (K_1, V_1) \rightarrow \text{List}[(K_2, V_2)]
\]

**예제: Word Count**

```python
def map(document_name: str, document_contents: str) -> List[Tuple[str, int]]:
    """
    입력: (문서 이름, 문서 내용)
    출력: [(단어, 1), (단어, 1), ...]
    """
    result = []
    for word in document_contents.split():
        result.append((word, 1))
    return result

# 예시
# map("doc1", "hello world hello")
# → [("hello", 1), ("world", 1), ("hello", 1)]
```

### 2. Reduce 함수

**정의**: 동일한 중간 키에 대한 모든 값을 집계

```
reduce(k2, list(v2)) → list(v2)
```

**수식 표현**:

\[
\text{Reduce}: (K_2, \text{List}[V_2]) \rightarrow \text{List}[V_2]
\]

**예제: Word Count**

```python
def reduce(word: str, counts: Iterator[int]) -> List[int]:
    """
    입력: (단어, [1, 1, 1, ...])
    출력: [총 개수]
    """
    total = sum(counts)
    return [total]

# 예시
# reduce("hello", [1, 1])
# → [2]
```

### 3. 함수형 프로그래밍과의 관계

MapReduce는 Lisp의 `map`과 `fold`(reduce)에서 영감을 받았다:

```
┌─────────────────────────────────────────────────────────────────┐
│           함수형 프로그래밍 vs MapReduce                          │
├─────────────────────────────────────────────────────────────────┤
│                                                                  │
│  Lisp (단일 머신):                                               │
│  ─────────────────                                               │
│  (map f list)           ; 각 원소에 f 적용                       │
│  (reduce g list)        ; 리스트를 단일 값으로 축소              │
│                                                                  │
│  MapReduce (분산 시스템):                                        │
│  ────────────────────────                                        │
│  Map Phase:   데이터를 여러 머신에서 병렬 변환                   │
│  Shuffle:     키별로 데이터 그룹화 (자동)                        │
│  Reduce Phase: 그룹화된 데이터 집계                              │
│                                                                  │
│  핵심 차이:                                                      │
│  ─────────                                                       │
│  • 데이터가 메모리에 없고 분산 파일시스템(GFS)에 있음            │
│  • 머신 간 데이터 이동(Shuffle)이 필요                           │
│  • 장애 복구 메커니즘 필요                                        │
│                                                                  │
└─────────────────────────────────────────────────────────────────┘
```

### 4. 타입 시그니처 (논문 원문)

논문에서 정의한 정형화된 타입:

```
map     : (k1, v1)        → list(k2, v2)
reduce  : (k2, list(v2))  → list(v2)
```

**중요**: Map의 출력 타입과 Reduce의 입력 타입이 일치해야 한다.

### 3. 태스크 상태 관리

논문에서 정의한 태스크 상태:

| 상태 | 설명 |
|------|------|
| **idle** | 아직 할당되지 않음 |
| **in-progress** | Worker에 할당되어 실행 중 |
| **completed** | 성공적으로 완료됨 |

**상태 전이**:

```
idle ──[assign]──► in-progress ──[complete]──► completed
                        │
                        └──[worker fail]──► idle (재실행)
```

### 4. 데이터 파티셔닝

**Map 파티셔닝 (M splits)**:
- 입력 파일을 M개의 조각으로 분할
- 각 조각은 보통 16MB~64MB (GFS 블록 크기)
- 각 조각에 대해 하나의 Map 태스크

**Reduce 파티셔닝 (R partitions)**:
- 사용자가 R 값 지정
- 파티션 함수: `hash(key) mod R`
- 각 파티션에 대해 하나의 Reduce 태스크

**수식**:

\[
\text{partition}(k) = \text{hash}(k) \mod R
\]

## Shuffle Phase 상세

### 1. 중간 데이터 처리

```
┌─────────────────────────────────────────────────────────────────┐
│                   Shuffle 단계 상세                               │
├─────────────────────────────────────────────────────────────────┤
│                                                                  │
│  Map Worker (local disk)              Reduce Worker              │
│  ────────────────────────             ─────────────              │
│                                                                  │
│  ┌─────────────────────┐                                         │
│  │ Map Output          │                                         │
│  │ ┌─────────────────┐ │   HTTP Pull                            │
│  │ │ Partition 0     │─┼─────────────────► Reduce 0             │
│  │ ├─────────────────┤ │                                         │
│  │ │ Partition 1     │─┼─────────────────► Reduce 1             │
│  │ ├─────────────────┤ │                                         │
│  │ │ Partition 2     │─┼─────────────────► Reduce 2             │
│  │ └─────────────────┘ │                                         │
│  └─────────────────────┘                                         │
│                                                                  │
│  처리 순서:                                                       │
│  1. Map이 출력을 R개 파티션으로 분할하여 로컬 디스크에 저장       │
│  2. Master에게 파일 위치 보고                                    │
│  3. Reduce Worker가 HTTP로 필요한 파티션 pull                    │
│  4. Reduce Worker가 받은 데이터를 키별로 정렬                     │
│                                                                  │
└─────────────────────────────────────────────────────────────────┘
```

### 2. 정렬 (Sorting)

Reduce 단계 전, 동일한 키를 가진 모든 값이 그룹화되어야 한다:

```python
# Shuffle 후 Reduce 입력 형태
# 정렬되어 같은 키가 연속으로 나타남

("apple", 1)
("apple", 1)
("apple", 1)
("banana", 1)
("banana", 1)
("cherry", 1)

# Reduce 함수에 전달되는 형태
reduce("apple", [1, 1, 1])   # Iterator
reduce("banana", [1, 1])
reduce("cherry", [1])
```

## Fault Tolerance

### 1. Worker 장애 복구

```
┌─────────────────────────────────────────────────────────────────┐
│                 Worker 장애 복구 메커니즘                         │
├─────────────────────────────────────────────────────────────────┤
│                                                                  │
│  Master의 역할:                                                  │
│  ──────────────                                                  │
│  • 모든 Worker에게 주기적으로 ping                               │
│  • 응답 없으면 (timeout) Worker를 dead로 표시                    │
│                                                                  │
│  Map Worker 장애 시:                                             │
│  ────────────────────                                            │
│  • 완료된 Map 태스크도 재실행 필요 (중간 결과가 로컬 디스크)     │
│  • 진행 중인 Map 태스크는 당연히 재실행                          │
│  • 새 Worker에게 재할당                                          │
│                                                                  │
│  Reduce Worker 장애 시:                                          │
│  ───────────────────────                                         │
│  • 완료된 Reduce 태스크는 재실행 불필요 (결과가 GFS)             │
│  • 진행 중인 Reduce 태스크만 재실행                              │
│                                                                  │
│  ┌──────────────────────────────────────────────────────────┐   │
│  │                                                          │   │
│  │   Map 출력 → 로컬 디스크 (휘발성)                        │   │
│  │   Reduce 출력 → GFS (영구 저장)                          │   │
│  │                                                          │   │
│  │   따라서 Map은 재실행, Reduce는 결과 보존                 │   │
│  │                                                          │   │
│  └──────────────────────────────────────────────────────────┘   │
│                                                                  │
└─────────────────────────────────────────────────────────────────┘
```

### 2. Master 장애 처리

논문에서는 Master 장애를 단순하게 처리:

```
Master 장애 시:
  → 전체 MapReduce 작업 중단
  → 클라이언트가 재시도
  
이유:
  • Master는 단일 프로세스 (SPOF)
  • Master 장애 확률은 낮음 (수천 Worker vs 1 Master)
  • 체크포인팅으로 복구 가능하나, 논문에서는 미구현
```

### 3. Stragglers 처리 (Backup Tasks)

**문제**: 일부 태스크가 비정상적으로 느림

**원인**:
- 디스크 오류로 읽기 속도 저하
- 다른 작업과 리소스 경쟁
- CPU 캐시 문제

**해결: Backup Task**

```
┌─────────────────────────────────────────────────────────────────┐
│                   Backup Task 메커니즘                            │
├─────────────────────────────────────────────────────────────────┤
│                                                                  │
│  MapReduce 작업이 거의 완료될 때:                                │
│                                                                  │
│  1. 아직 진행 중인 태스크들을 다른 Worker에도 할당              │
│  2. 두 복사본 중 하나라도 완료되면 태스크 완료 처리              │
│                                                                  │
│  ┌────────────────────────────────────────────────────────┐     │
│  │                                                        │     │
│  │  Task T가 느림 (Worker A)                              │     │
│  │         │                                              │     │
│  │         ├──► 같은 Task T를 Worker B에도 할당            │     │
│  │         │                                              │     │
│  │  Worker A: ████████████░░░░░░░░ (70%)                 │     │
│  │  Worker B: ████████████████████ (100%, 먼저 완료)      │     │
│  │         │                                              │     │
│  │         └──► Task T 완료, Worker A의 작업 취소         │     │
│  │                                                        │     │
│  └────────────────────────────────────────────────────────┘     │
│                                                                  │
│  효과 (논문 실험):                                               │
│  • Backup 없이: 44% 더 오래 걸림                                 │
│  • Backup 비용: 전체 연산량의 몇 % 추가                          │
│                                                                  │
└─────────────────────────────────────────────────────────────────┘
```

## 최적화 기법

### 1. Combiner 함수

**목적**: Map 출력을 Reduce로 전송하기 전에 로컬에서 부분 집계

```
┌─────────────────────────────────────────────────────────────────┐
│                    Combiner 최적화                                │
├─────────────────────────────────────────────────────────────────┤
│                                                                  │
│  Without Combiner:                                               │
│  ─────────────────                                               │
│  Map 1: ("hello", 1), ("hello", 1), ("world", 1)                │
│  Map 2: ("hello", 1), ("hello", 1)                              │
│          │                                                       │
│          └──► 5개 레코드를 네트워크로 전송                       │
│                                                                  │
│  With Combiner:                                                  │
│  ───────────────                                                 │
│  Map 1: ("hello", 1), ("hello", 1), ("world", 1)                │
│         │                                                        │
│         └──► Combiner 적용                                       │
│              ("hello", 2), ("world", 1)                         │
│                                                                  │
│  Map 2: ("hello", 1), ("hello", 1)                              │
│         │                                                        │
│         └──► Combiner 적용                                       │
│              ("hello", 2)                                        │
│          │                                                       │
│          └──► 3개 레코드만 네트워크로 전송                       │
│                                                                  │
└─────────────────────────────────────────────────────────────────┘
```

**조건**: Reduce 함수가 교환법칙과 결합법칙을 만족해야 함

```python
# Combiner 적용 가능 (결합법칙, 교환법칙 만족)
def reduce_sum(key, values):
    return sum(values)

# Combiner 적용 불가 (평균은 결합법칙 불만족)
def reduce_average(key, values):
    return sum(values) / len(values)
```

### 2. Partitioner 커스터마이징

기본 파티션 함수: `hash(key) mod R`

커스텀 파티셔너 예제:

```python
# URL의 호스트별로 파티셔닝
def url_partitioner(url: str, num_partitions: int) -> int:
    host = urlparse(url).hostname
    return hash(host) % num_partitions

# 효과: 같은 호스트의 URL이 같은 Reduce 태스크로
```

## 의미론적 보장

### 1. Deterministic Functions

```
Map과 Reduce가 결정론적(deterministic)일 때:
  → 전체 MapReduce 연산은 순차 실행과 동일한 결과
  → 장애 복구 후에도 동일한 결과 보장
```

### 2. Non-deterministic Functions

```
Map이나 Reduce가 비결정론적일 때:
  → 약한 보장 제공
  → 각 Reduce의 출력은 비결정론적 순차 실행의 
    어떤 한 실행 결과와 동일
```

### 3. Atomic Commits

```
┌─────────────────────────────────────────────────────────────────┐
│                   Atomic Commit 메커니즘                          │
├─────────────────────────────────────────────────────────────────┤
│                                                                  │
│  Map 태스크 완료 시:                                             │
│  ─────────────────                                               │
│  1. 출력을 임시 파일에 작성                                      │
│  2. Master에게 완료 메시지 (임시 파일 위치 포함)                 │
│  3. Master가 이미 완료된 Map이면 무시 (중복 방지)                │
│                                                                  │
│  Reduce 태스크 완료 시:                                          │
│  ───────────────────                                             │
│  1. 출력을 임시 파일에 작성                                      │
│  2. GFS의 atomic rename으로 최종 파일로 이동                     │
│  3. rename은 atomic → 부분 결과 노출 없음                        │
│                                                                  │
└─────────────────────────────────────────────────────────────────┘
```

## 영향과 후속 연구

### 1. 오픈소스 구현

| 프로젝트 | 연도 | 특징 |
|---------|------|------|
| **Hadoop** | 2006 | 가장 널리 사용된 구현, HDFS + MapReduce |
| **Disco** | 2008 | Erlang 기반, Nokia 개발 |
| **Spark** | 2012 | In-memory 처리, MapReduce 대체 |

### 2. MapReduce의 한계와 Spark의 등장

```
┌─────────────────────────────────────────────────────────────────┐
│                MapReduce 한계 vs Spark 개선                       │
├─────────────────────────────────────────────────────────────────┤
│                                                                  │
│  MapReduce 한계:                                                 │
│  ───────────────                                                 │
│  • 중간 결과를 항상 디스크에 저장 → I/O 병목                     │
│  • 반복 알고리즘에 비효율적 (매번 디스크 읽기/쓰기)              │
│  • 복잡한 파이프라인 구성 어려움                                  │
│                                                                  │
│  Spark 개선:                                                     │
│  ────────────                                                    │
│  • RDD (Resilient Distributed Dataset): 메모리 캐싱             │
│  • 지연 평가 (Lazy Evaluation)                                   │
│  • DAG 스케줄러: 복잡한 파이프라인 최적화                        │
│  • 반복 알고리즘 10-100배 성능 향상                              │
│                                                                  │
│  성능 비교 (논문 "Spark: Cluster Computing with Working Sets"):  │
│  ────────────────────────────────────────────────────────────   │
│  Logistic Regression (반복 알고리즘):                            │
│    • Hadoop: 127초/iteration                                     │
│    • Spark:  0.9초/iteration (140배 빠름)                        │
│                                                                  │
└─────────────────────────────────────────────────────────────────┘
```

### 3. 현대적 대안들 비교

| 시스템 | 처리 모델 | 데이터 저장 | 적합 용도 |
|--------|----------|------------|----------|
| **MapReduce** | Batch | Disk | 대규모 일괄 처리 |
| **Spark** | Batch + Stream | Memory | 반복 알고리즘, 대화형 분석 |
| **Flink** | Stream-first | Memory | 실시간 스트리밍 |
| **Dataflow** | Unified | Memory | 배치/스트림 통합 |

### 4. MapReduce의 지속적 영향

```
MapReduce 논문의 영향:
  
1. 분산 처리의 대중화
   → 이전: 전문가만 가능
   → 이후: 간단한 함수 정의로 분산 처리

2. 장애 복구 자동화
   → 개발자가 장애 처리 코드 작성 불필요

3. 수평 확장 모델 확립
   → "더 빠른 머신" 대신 "더 많은 머신"

4. 빅데이터 생태계 기반
   → Hadoop, Spark, Hive, Pig 등의 기반
```
