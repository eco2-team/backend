# #37: 왜 Consistent Hash + StatefulSet이 '최종해'가 아니었나

> **시리즈**: SSE 아키텍처 진화 (Part 6)  
> **작성일**: 2025-12-27  
> **이전 글**: [#98 SSE Gateway Evolution](https://rooftopsnow.tistory.com/98)  
> **다음 글**: [#34 SSE HA Architecture](./34-sse-HA-architecture.md)

---

## TL;DR

[#98](https://rooftopsnow.tistory.com/98)에서 Consistent Hash + StatefulSet으로 "연결 라우팅"을 해결했지만, 
**이벤트가 해당 Pod로 도착하는 것은 보장되지 않았다**.

Worker의 해싱과 Istio의 해싱이 일치하지 않아 결국 **Fan-out 계층**이 필요해졌다.

---

## 1. 98번 포스팅 핵심 결론 (10줄 요약)

| Phase | 모델 | 결과 |
|-------|------|------|
| 1 | 연결당 XREAD | 50 VU → CPU 85%, 완료율 62% |
| 2 | 단일 Consumer | CPU 15%, 수평확장 불가 |
| 3 | StatefulSet + Consistent Hash | **해싱 정합성 불일치** |

상세 구현과 성능 지표는 [#98 포스팅](https://rooftopsnow.tistory.com/98) 참조.

**핵심 문제**: Phase 3에서 Istio는 연결을 올바른 Pod로 라우팅하지만,  
Worker가 이벤트를 쓰는 shard와 Pod가 읽는 shard가 **일치하지 않았다**.

---

## 2. 본질적 문제: "연결 라우팅" ≠ "이벤트 라우팅"

### 2.1 Istio가 보장하는 것

Istio Consistent Hash가 보장하는 것:
- ✅ **같은 job_id 요청 → 같은 Pod로 연결**
- ❌ **해당 job_id의 이벤트가 그 Pod의 shard로 도착** (보장 안 됨)

```
┌─────────────────────────────────────────────────────────────────────────┐
│ Istio는 연결만 고정, 이벤트 라우팅은 비어있다                              │
├─────────────────────────────────────────────────────────────────────────┤
│                                                                         │
│                    ┌──────────────────────────────────────────┐        │
│                    │              Istio                        │        │
│                    │         Consistent Hash                   │        │
│                    │                                           │        │
│  Client ─────────▶ │  hash("job-123") → Pod-1 선택             │        │
│  (job-123)         │                                           │        │
│                    └───────────────────────┬──────────────────┘        │
│                                            │                            │
│                                            ▼                            │
│                                   ┌─────────────┐                       │
│                                   │ Pod-1       │                       │
│                                   │ shard_id=1  │                       │
│                                   │             │                       │
│                                   │ XREAD       │                       │
│                                   │ scan:events:1                       │
│                                   └─────────────┘                       │
│                                            ▲                            │
│                                            │ ❌ 이벤트 없음!            │
│                                            │                            │
│  ─ ─ ─ ─ ─ ─ ─ ─ ─ ─ ─ ─ ─ ─ ─ ─ ─ ─ ─ ─ ─│─ ─ ─ ─ ─ ─ ─ ─ ─ ─ ─ ─   │
│                                            │                            │
│                                   ┌─────────────┐                       │
│                                   │   Worker    │                       │
│                                   │             │                       │
│                                   │ MD5("job-123")                      │
│                                   │ % 4 = 3     │                       │
│                                   │             │                       │
│                                   │ XADD        │                       │
│                                   │ scan:events:3  ◀── 여기에 씀!       │
│                                   └─────────────┘                       │
│                                                                         │
│  ⚠️ Worker의 해싱(MD5 % shard_count)과                                  │
│     Istio의 해싱(Maglev/Ring Hash)이 일치하지 않음                       │
│                                                                         │
│  결과: Client는 Pod-1에 연결됐지만,                                      │
│       이벤트는 shard-3에 있어서 영원히 수신 못 함                         │
│                                                                         │
└─────────────────────────────────────────────────────────────────────────┘
```

### 2.2 해싱 알고리즘 불일치

| 컴포넌트 | 해싱 방식 | 결과 |
|----------|-----------|------|
| **Worker (Python)** | `MD5(job_id) % shard_count` | 정수 0~3 |
| **Istio (Envoy)** | Maglev/Ring Hash | 가용 Pod 중 선택 |

두 해싱 결과가 **우연히 일치할 확률**: shard_count=4, Pod=4일 때 약 **25%**  
→ 나머지 **75%는 이벤트 누락**

## 3. HPA/KEDA와의 철학 충돌

StatefulSet + shard 고정 모델의 근본적 문제:

```
┌─────────────────────────────────────────────────────────────────────────┐
│ HPA/KEDA 스케일링 시 문제                                                │
├─────────────────────────────────────────────────────────────────────────┤
│                                                                         │
│  Before: 4 Pods (shard 0, 1, 2, 3)                                     │
│                                                                         │
│  sse-gateway-0 ──▶ scan:events:0                                       │
│  sse-gateway-1 ──▶ scan:events:1                                       │
│  sse-gateway-2 ──▶ scan:events:2                                       │
│  sse-gateway-3 ──▶ scan:events:3                                       │
│                                                                         │
│  ═══════════════════════════════════════════════════════════════════   │
│                                                                         │
│  After: HPA가 6 Pods로 확장                                             │
│                                                                         │
│  sse-gateway-0 ──▶ scan:events:0                                       │
│  sse-gateway-1 ──▶ scan:events:1                                       │
│  sse-gateway-2 ──▶ scan:events:2                                       │
│  sse-gateway-3 ──▶ scan:events:3                                       │
│  sse-gateway-4 ──▶ scan:events:??? ◀── shard_count=4면 어디 읽어야?    │
│  sse-gateway-5 ──▶ scan:events:??? ◀── 새 Pod는 놀게 됨                │
│                                                                         │
│  ⚠️ shard_count를 동적으로 바꾸면?                                       │
│     → 기존 해시 결과가 바뀜 → 기존 연결이 잘못된 Pod로 감                 │
│                                                                         │
│  결론: Pod 수 변동 = shard 재할당 필요 = 정합성 문제                      │
│                                                                         │
└─────────────────────────────────────────────────────────────────────────┘
```

## 4. 근본 원인: 연결과 이벤트 소비가 결합됨

```
┌────────────────┐
│   SSE Pod      │
│                │
│  1. 연결 수신  │ ◀── Istio가 결정
│       +        │
│  2. 이벤트 소비│ ◀── 자신의 shard만 읽음
│       +        │
│  3. 클라이언트 │
│     전달       │
└────────────────┘

문제: 1번과 2번이 같은 곳에서 일어나야 함
    → 외부(Istio)와 내부(shard)의 매핑을 강제해야 함
    → Pod 수 변경 시 매핑 재조정 필요
    → HPA/KEDA 사용 불가
```

## 5. 해결책: 라우팅(연결)과 소비(이벤트) 분리

**핵심 인사이트**: SSE Pod는 이벤트를 직접 읽지 않는다.

```
┌─────────────────────────────────────────────────────────────────────────┐
│ 분리된 모델: Fan-out 계층 도입                                           │
├─────────────────────────────────────────────────────────────────────────┤
│                                                                         │
│  ┌────────────────┐     ┌────────────────┐     ┌────────────────┐      │
│  │  SSE Pod 0     │     │  SSE Pod 1     │     │  SSE Pod 2     │      │
│  │  연결 수신만   │     │  연결 수신만   │     │  연결 수신만   │      │
│  │  이벤트 구독   │     │  이벤트 구독   │     │  이벤트 구독   │      │
│  └───────┬────────┘     └───────┬────────┘     └───────┬────────┘      │
│          │ SUBSCRIBE            │ SUBSCRIBE            │ SUBSCRIBE     │
│          ▼                      ▼                      ▼               │
│  ┌─────────────────────────────────────────────────────────────────┐   │
│  │                      Redis Pub/Sub                               │   │
│  │  sse:events:job-123    sse:events:job-456    sse:events:job-789 │   │
│  └────────────────────────────────▲────────────────────────────────┘   │
│                                   │ PUBLISH                            │
│  ┌─────────────────────────────────────────────────────────────────┐   │
│  │                      Event Router                                │   │
│  │  XREADGROUP (Consumer Group)                                    │   │
│  │  모든 shard 소비 → job_id별 채널로 발행                          │   │
│  └────────────────────────────────▲────────────────────────────────┘   │
│                                   │ XADD                               │
│  ┌─────────────────────────────────────────────────────────────────┐   │
│  │                      Redis Streams                               │   │
│  │  scan:events:0   scan:events:1   scan:events:2   scan:events:3  │   │
│  └─────────────────────────────────────────────────────────────────┘   │
│                                   ▲ Worker                             │
│                                                                        │
│  ✅ SSE Pod는 어느 shard를 읽는지 몰라도 됨                             │
│  ✅ Event Router가 모든 이벤트를 수집 → 올바른 채널로 분배              │
│  ✅ Pod 수와 shard 수가 독립적 → HPA/KEDA 자유롭게 사용 가능           │
│                                                                        │
└─────────────────────────────────────────────────────────────────────────┘
```

## 6. 결론

| 관점 | StatefulSet + Consistent Hash | Fan-out 계층 |
|------|------------------------------|--------------|
| **연결 라우팅** | ✅ 일관된 Pod 선택 | ✅ 아무 Pod나 OK |
| **이벤트 도달 보장** | ❌ 해싱 불일치 | ✅ Pub/Sub로 분배 |
| **HPA/KEDA** | ❌ shard 재할당 필요 | ✅ Pod 수 무관 |
| **운영 복잡도** | 높음 (Envoy Filter, StatefulSet) | 낮음 (Deployment) |

**최종 결론**:

> **"연결 라우팅(Istio)과 이벤트 소비(Streams)를 분리하면,**  
> **SSE Pod는 Stateless가 되고, HPA/KEDA로 자유롭게 확장할 수 있다."**
