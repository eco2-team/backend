# 기술 리포트 작성 가이드라인

기술 리포트 작성 시 준수해야 할 스타일 가이드입니다.

## 1. 언어 및 문체

### 존칭 사용 (Formal Korean)

기술 리포트는 공식 문서이므로 존칭체를 사용합니다.

| 비권장 | 권장 |
|--------|------|
| ~이다 | ~입니다 |
| ~한다 | ~합니다 |
| ~된다 | ~됩니다 |
| ~했다 | ~하였습니다 |
| ~없다 | ~없습니다 |
| ~있다 | ~있습니다 |

**예시**:
- (X) "메시지가 PEL에서 제거된다"
- (O) "메시지가 PEL에서 제거됩니다"

### 능동태 선호

가능한 한 능동태를 사용하여 명확성을 높입니다.

- (X) "메시지가 처리되었습니다"
- (O) "Consumer가 메시지를 처리하였습니다"

### 기술 용어 일관성

동일한 개념에 대해 일관된 용어를 사용합니다. 최초 사용 시 정의를 제공합니다.

## 2. 비교 표현

### AS-IS / TO-BE 용어 사용

변경 전/후를 표현할 때 "Before/After" 대신 "AS-IS/TO-BE"를 사용합니다.

| 비권장 | 권장 |
|--------|------|
| Before | AS-IS |
| After | TO-BE |
| 수정 전 | AS-IS |
| 수정 후 | TO-BE |
| 변경 전 | AS-IS |
| 변경 후 | TO-BE |

**코드 블록 예시**:
```python
# AS-IS: 실패해도 ACK 실행
await self._redis.xack(...)

# TO-BE: 성공한 경우만 ACK
if success:
    await self._redis.xack(...)
```

**ASCII 다이어그램 예시**:
```
┌─────────────────────────────────────────────┐
│  [AS-IS - 문제 상황]                         │
│    ...                                       │
├─────────────────────────────────────────────┤
│  [TO-BE - 개선된 상황]                       │
│    ...                                       │
└─────────────────────────────────────────────┘
```

## 3. 시각화

### 다이어그램 아래 설명 필수

모든 시각화(ASCII 다이어그램, 플로우차트 등) 아래에 반드시 텍스트 설명을 추가합니다.

**올바른 형식**:
```
[ASCII 다이어그램]

Event Router는 Worker가 Redis Streams에 발행한 이벤트를 Consumer가 수신하여
Processor를 통해 처리합니다. Processor는 Lua Script를 활용하여 원자적인
State 갱신과 멱등성을 보장하며...
```

### 설명 작성 원칙

1. **요약 우선**: 다이어그램의 핵심 흐름을 1-2문장으로 요약합니다
2. **세부 설명**: 중요한 컴포넌트나 결정 포인트를 상세히 설명합니다
3. **의미 부여**: 왜 이 구조가 중요한지, 어떤 문제를 해결하는지 설명합니다

### 다이어그램 종류별 가이드

| 종류 | 용도 | 필수 설명 항목 |
|------|------|----------------|
| 아키텍처 다이어그램 | 전체 구조 표현 | 각 컴포넌트 역할, 데이터 흐름 |
| 시퀀스 다이어그램 | 시간순 흐름 표현 | 각 단계의 목적, 실패 시나리오 |
| 비교 다이어그램 | AS-IS/TO-BE 비교 | 변경 이유, 개선 효과 |

## 4. 문서 구조

### 권장 섹션 구성

1. **Executive Summary**: 핵심 내용 1-2 문단 요약
2. **아키텍처 개요**: 시각화와 함께 전체 구조 설명
3. **Bugs (수정 완료)**: 발견된 버그와 수정 내용
4. **Plans (개선 완료)**: 계획된 개선 사항과 구현 결과
5. **Plans (미적용)**: 추후 진행 예정인 사항
6. **기존 양호 구현**: 잘 설계된 부분 인정
7. **구현 결과 요약**: 완료/미완료 항목 테이블
8. **변경 파일 목록**: 수정된 파일과 변경 내용

### 버그/개선 항목 형식

```markdown
### 3.1 [P0 Critical] 항목 제목 ✅ Fixed

**위치**: `파일명.py:라인번호`

| 항목 | 내용 |
|------|------|
| **증상** | 사용자 관점에서 관찰되는 문제 |
| **영향** | 비즈니스/시스템에 미치는 영향 |
| **원인** | 기술적 근본 원인 |

**수정 내용**:
[코드 블록 또는 설명]

**커밋**: `해시값` 커밋 메시지
```

- [ ] 모든 다이어그램 아래에 설명이 있는가?
